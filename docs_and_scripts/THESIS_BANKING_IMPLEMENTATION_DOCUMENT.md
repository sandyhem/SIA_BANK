# Banking Platform Implementation Thesis Document

## Abstract
This document presents the implementation narrative of a secure banking platform built using a microservices architecture, with a focus on the complete customer lifecycle, role-based operational interfaces, and defense-in-depth security. The platform is designed around three domain microservices, namely Auth Service, Account Service, and Transaction Service, integrated behind an NGINX proxy layer that acts as the transport gateway and TLS termination boundary. The implementation follows a regulated banking flow in which a user first establishes an application identity, then evolves into a valid customer profile through KYC governance, and finally becomes eligible for account operations and digital payments. In addition to application-level security controls, the system introduces post-quantum and hybrid cryptographic readiness, mTLS-oriented service trust boundaries, and kernel-level runtime observability using eBPF with Prometheus and Grafana for operational and security analytics.

## System Context and Architectural Intent
The architectural intent of this platform is to separate business responsibility by bounded context while preserving a single banking journey for the end user. The Auth Service is responsible for identity, role assignment, and customer/KYC metadata governance. The Account Service is responsible for productized banking accounts, balances, and account state transitions. The Transaction Service is responsible for value movement, transfer validation, and transactional history. These services are exposed through an NGINX reverse proxy configuration, where NGINX serves as the traffic entry point and TLS control plane. This design allows independent scaling, isolated failure domains, and cleaner security controls. The platform is also designed for cryptographic evolution, where conventional protections can be combined with post-quantum mechanisms in a hybrid model to reduce future cryptographic risk while retaining practical interoperability.

## Module-Wise Banking Implementation

### 1. Identity and Access Module (Auth Service)
The first operational module in the banking lifecycle is identity creation. A prospective customer begins as a platform user through a registration transaction, where core credentials and profile attributes are captured and normalized. At this stage, the objective is not yet to provide full banking privileges, but to establish a strongly attributable identity with role semantics such as USER or ADMIN. The Auth Service persists this identity and exposes authenticated entry for future operations. In implementation terms, this module is the control point for user login, registration, role-aware policy decisions, and authentication response payloads that are consumed by the frontend channels.

From a thesis perspective, this stage is critical because it introduces the distinction between authentication identity and regulated banking identity. A registered user is not immediately an active banking customer. This gap is intentional and mirrors real financial governance, where legal and compliance verification must occur before any monetary product is activated. Therefore, the Auth Service works as the policy root and eligibility gatekeeper for all downstream banking capabilities.

### 2. Customer Profile and KYC Governance Module (Auth Service)
After identity creation, the platform transitions the user into customer onboarding by generating a customer record (for example, CIF-oriented customer identity) and initializing KYC as a controlled state machine. The implementation follows typical statuses such as pending, under review, verified, and rejected. During this phase, the customer may have profile presence in the system but still remain non-operational for account activation or payment privileges. This distinction ensures that regulatory controls are preserved as first-class backend rules instead of optional UI checks.

The KYC lifecycle is directly controlled by administrative authority through the ADMIN interface. An admin reviews identity artifacts and compliance metadata and can approve or reject the customer. Once approved, the customer becomes a verified customer, and only then is the user considered eligible for banking product activation. This module therefore bridges legal trust and technical trust: legal trust is achieved through KYC verification, while technical trust is reflected in service-level permission to proceed with account creation and transaction enablement.

### 3. Account Product Module (Account Service)
The Account Service owns the bank account domain, including account creation, account metadata, status, and balances. In the implemented flow, account opening is tightly coupled with customer eligibility. A verified customer can request account opening, and the service generates account records that become the principal financial instruments for further operations. This design prevents unauthorized account fabrication and keeps account issuance aligned with compliance state.

Once an account exists, the customer can view account details and balance in the home interface. The balance view is not treated as a trivial display operation; in this implementation it is integrated with user-facing trust controls and role-aware behavior. The account module also supports operational constraints such as account status and customer linkage, so that account-level operations cannot be detached from verified customer context.

### 4. Payments and Ledger Activity Module (Transaction Service)
The Transaction Service executes monetary movement and provides transaction history as an auditable user-facing and operator-facing capability. The implemented send-money flow validates essential constraints such as source account integrity and operation eligibility before transfer execution. The user journey is intentionally confirmation-driven, where the transfer requires explicit user action and security confirmation via MPIN in the client flow. This mirrors retail banking ergonomics where payment intent must be confirmed by a knowledge factor.

After transfer completion, transaction history becomes the narrative record of financial activity. The system exposes recent and complete transaction views so that users can reconcile account behavior and verify outcomes. From a thesis standpoint, this module demonstrates separation of concerns: account ownership and balance are kept in Account Service, while value-transfer semantics and transaction history remain in Transaction Service, reducing coupling and preserving a cleaner financial domain model.

## Interface Model: User Interface and Admin Interface
The platform provides two operational interfaces mapped to role semantics.

The USER interface is oriented toward day-to-day banking tasks. The user registers and logs in, completes profile onboarding, waits for compliance approval, opens an account after verification, views balance on the home page, reviews transaction history, and initiates send-money operations. The user interface thus represents the customer journey from digital identity to active transactional banking.

The ADMIN interface is governance-centric rather than transaction-centric. The admin reviews and decides KYC outcomes, thereby controlling the eligibility boundary for account activation. By separating this interface from regular user functions, the platform reduces accidental privilege overlap and reinforces least-privilege operations. In practical terms, the admin role transforms a pending customer into a verified customer, and that state transition unlocks core banking capabilities for the customer.

## End-to-End Customer Lifecycle Narrative
A complete lifecycle begins when a new individual creates a user account in the platform. At this point, the person has a valid digital login identity but is not yet a fully active banking customer. The person then submits customer profile data, creating a regulated customer record with pending KYC state. An administrator reviews the KYC context and approves it when regulatory requirements are satisfied. At this exact boundary, the person becomes a verified customer. Following verification, the customer can open a bank account through the account module. Once the account is opened, the home page displays account and balance context, transaction history can be viewed for transparency, and send-money becomes available as an operational capability guarded by confirmation controls including MPIN in the transfer flow.

This sequence is important academically because it demonstrates process integrity across identity, compliance, account productization, and payment execution without collapsing governance boundaries. It also shows how technical implementation can embody real banking control flows rather than only simulating basic CRUD operations.

## Microservices Behind NGINX Proxy and TLS Termination
The deployed topology places Auth, Account, and Transaction microservices behind NGINX. NGINX serves as reverse proxy and TLS termination layer at the edge, providing centralized ingress control, certificate handling, and routing policy. In this model, clients do not need to manage service-specific transport paths directly; instead, NGINX enforces a controlled entry point and forwards requests to the appropriate backend service.

The implementation objective is to support trust continuity from edge to service boundaries. TLS at the proxy provides secure client ingress, while service-to-service trust can be extended through mTLS patterns and hardened internal transport policies. This layered model is practical for production systems because it allows independent service development while preserving uniform security posture at ingress and east-west communication boundaries.

## Hybrid Post-Quantum Cryptography and mTLS Orientation
The platform incorporates post-quantum readiness through hybrid cryptographic integration, including support paths based on ML-DSA and ML-KEM concepts in the authentication domain. The broader strategy is not merely algorithm replacement but algorithm coexistence, where classical and post-quantum mechanisms can be used in a hybrid mode during migration phases. This approach reduces immediate interoperability risk while building resistance against future cryptanalytic advances.

In a thesis framing, hybrid post-quantum adoption is valuable because it reflects realistic modernization strategy: systems cannot always switch cryptographic stacks abruptly, especially in financial ecosystems with diverse clients and dependencies. By using a migration-safe hybrid stance and maintaining mTLS-oriented service trust boundaries, the architecture aligns current operational compatibility with long-horizon cryptographic resilience.

## Runtime Security and Observability: eBPF, Prometheus, and Grafana
Beyond cryptography and role controls, the implementation introduces runtime security observability using eBPF. eBPF enables kernel-level telemetry collection without modifying microservice application code. This provides low-overhead visibility into live communication metadata, process-level networking behavior, and potentially anomalous interaction patterns among services. In a microservices banking platform, this capability is valuable for detecting suspicious lateral communication, unusual flow bursts, and service behavior that departs from expected topologies.

The telemetry pipeline is integrated with Prometheus for time-series metric collection and with Grafana for visualization and operational dashboards. Prometheus acts as the metric persistence and query layer, while Grafana turns these metrics into actionable dashboards for SRE and security operations. Together, eBPF, Prometheus, and Grafana form an observability triad: eBPF captures runtime truth at kernel depth, Prometheus structures and stores it, and Grafana exposes it for diagnosis, trend analysis, and incident response.

## Security and Governance Positioning
The combined architecture demonstrates a layered security model. Identity and role control are handled in Auth Service. Regulatory trust is enforced via admin-controlled KYC transitions. Product and monetary state are separated into account and transaction domains. Transport security is centralized and standardized through NGINX TLS handling with mTLS-oriented trust posture for internal communications. Cryptographic modernization is addressed through hybrid post-quantum capability. Runtime observability is achieved via eBPF instrumentation with Prometheus/Grafana operational intelligence.

This layered approach is crucial in banking systems because no single control is sufficient. Authentication without KYC does not provide regulatory eligibility. Encryption without runtime monitoring does not provide behavioral visibility. Role checks without domain separation increase blast radius. The implementation therefore demonstrates a defense-in-depth pattern where each layer compensates for the limitations of the others.

## Conclusion
The implemented banking platform can be understood as a full lifecycle banking system rather than a simple transaction application. It begins with user identity, matures into verified customer status through admin-governed KYC, proceeds to account opening in a dedicated account domain, and culminates in secured money movement and transaction visibility in a dedicated transaction domain. The two-interface model cleanly separates customer actions from compliance authority. The three-microservice architecture behind NGINX provides modularity and controlled ingress. The hybrid post-quantum and mTLS-oriented security posture positions the system for long-term cryptographic resilience. The eBPF, Prometheus, and Grafana stack adds runtime intelligence and security observability that is essential for modern financial operations.

In thesis terms, this implementation demonstrates how banking business process integrity, microservice architecture, cryptographic modernization, and runtime observability can be integrated into a coherent engineering framework that is both academically structured and operationally practical.
